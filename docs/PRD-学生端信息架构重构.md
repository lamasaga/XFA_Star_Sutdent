# 新府学「一生一案」学生端信息架构重构 PRD

> **文档定位**：针对当前学生端信息杂乱、层级混乱、导航缺失的问题，重新设计信息架构、导航体系与页面职责。供前端重构与产品验收使用。  
> **读者**：前端工程师、产品设计师、测试  
> **对齐文档**：`CLAUDE.md`（项目总览）· `ROADMAP.md`（长期规划）· 现有代码  
> **最后更新**：2026-06-01

---

## 一、问题诊断

### 1.1 导航缺失：学生在页面间"迷路"

| 现状 | 问题 |
|------|------|
| `app/(student)/layout.tsx` 仅做权限校验，零导航 UI | 学生进入任意页面后，没有固定导航栏返回或切换页面 |
| 页面间跳转依赖 Dashboard 的「快捷入口」卡片 | 一旦离开 Dashboard，就"被困"在当前页面，无法直达其他页面 |
| 无面包屑、无当前位置指示 | 学生不知道自己在哪个页面，也看不清页面层级关系 |

### 1.2 信息杂乱：每个页面都想"大而全"

| 页面 | 当前内容 | 问题 |
|------|----------|------|
| **Dashboard** | 五维雷达 + 成绩趋势 + 快捷入口 + 评语 + 里程碑 + 活动 | 6 类信息平铺，首屏塞满，学生不知道先看什么 |
| **Profile** | 个人信息 + 统计卡片 + 五维雷达 + 评语 + 里程碑 + 活动 + 时间线 | 比 Dashboard 更全，与 Dashboard 大量重复 |
| **Space** | 统计卡片 + 心理测评 + 测评结果 + 心情趋势 + 今日心情 + 最近记录 + 空间解锁 | 心理测评和心情日记混在一个页面，功能边界模糊 |
| **Scores** | 统计卡片 + 学科均衡雷达 + 最新成绩柱状图 + 趋势图 + 明细表 | 相对清晰，但统计卡片和图表信息有重叠 |

**核心矛盾**：Dashboard 和 Profile 都是"全聚合页"，功能互相覆盖，学生分不清两者的区别。

### 1.3 层级混乱：同一信息出现在多个页面

| 信息 | 出现位置 | 问题 |
|------|----------|------|
| 五维雷达 | Dashboard、Profile | 两处显示，维护成本高，学生困惑 |
| 最新评语 | Dashboard、Profile | Dashboard 显示 2 条，Profile 显示 6 条，无清晰分层 |
| 里程碑列表 | Dashboard、Profile | 无差异化 |
| 活动记录 | Dashboard、Profile | 无差异化 |
| 心理测评 | Space、Assessments | Space 内嵌测评入口，Assessments 是独立完整页，功能重复 |

### 1.4 使用体验差：缺少"节奏感"

- **没有每日首屏**：学生打开系统应该看到"今天有什么待办"，而不是直接看到五维雷达
- **没有进度主线**：成长是线性的，但当前页面没有"接下来该做什么"的引导
- **没有搜索/全局入口**：想找一条旧评语或一次旧成绩，需要逐个页面翻

## 二、重构原则

### 2.1 "一个信息只出现在一个页面"

五维雷达只放在 **Profile**（档案总览），Dashboard 只放五维的**精简分数卡片**（无图表）。评语只放在 Profile，Dashboard 只放**最新一条评语的摘要**。

### 2.2 "Dashboard 是今日工作台，不是总档案"

Dashboard = 待办提醒 + 今日心情 + 本周打卡 + 快捷入口。  
Profile = 完整成长档案（所有历史信息的总览）。

### 2.3 "每个页面有且只有一个核心职责"

| 页面 | 核心职责 |
|------|----------|
| Dashboard | 今日工作台：待办、打卡、快捷入口 |
| Profile | 成长档案总览：个人信息 + 五维雷达 + 时间线 |
| Scores | 成绩分析：明细 + 趋势 + 学科均衡 |
| Assessments | 心理/职业测评：量表列表 + 历史结果 |
| Mood | 心情日记：记录 + 趋势 + 日历热力图 |
| Milestones | 里程碑：申报 + 审核状态 + 历史列表 |
| Activities | 活动记录：列表 + 积分 + 分类筛选 |
| Semester Reports | 学期报告：学期总结 + 教师评语汇总 |
| Settings | 个人设置：密码、通知、隐私偏好 |

### 2.4 "统一导航，全局可达"

所有学生端页面共享一个**固定侧边栏导航**，任何页面都能直接跳转到其他页面。导航支持当前页面高亮。

## 六、布局适配

### 6.1 桌面端布局

```
┌──────────────────────────────────────────────────────────┐
│  240px 侧边栏  │  主内容区（剩余宽度）                      │
│                │                                          │
│  [导航]        │  [面包屑]                                │
│                │  [页面标题]                              │
│                │  [页面内容]                              │
│                │                                          │
│  [用户信息]    │                                          │
└──────────────────────────────────────────────────────────┘
```

- 侧边栏固定，`position: fixed`，`z-index: 40`
- 主内容区 `margin-left: 240px`，`padding: 24px 32px`
- 最大内容宽度 `max-width: 1200px`，居中（防止大屏上内容过宽）

### 6.2 移动端布局

```
┌────────────────────────────────────────┐
│  ☰ 一生一案              👤 张明 Lv.5  │  ← 顶部导航栏
├────────────────────────────────────────┤
│  [页面标题]                              │
│  [页面内容]                              │
│                                        │
│  [底部 Tab 导航]                         │
│  首页 · 档案 · 成绩 · 更多              │
└────────────────────────────────────────┘
```

- 顶部：汉堡菜单 + 品牌名 + 用户简要信息
- 点击汉堡菜单 → 左侧滑出导航抽屉（全屏覆盖）
- 底部 Tab 导航（固定）：首页 / 档案 / 成绩 / 更多（展开其余页面）
- 主内容区 `padding: 16px`

### 6.3 响应式断点

| 断点 | 布局 | 侧边栏 | 底部导航 |
|------|------|--------|----------|
| `≥1024px` | 桌面 | 固定展开 | 无 |
| `768px~1023px` | 平板 | 可折叠（图标+文字） | 无 |
| `<768px` | 手机 | 抽屉（汉堡菜单触发） | 固定底部 Tab |

## 七、数据流调整

### 7.1 删除冗余 API 调用

当前 Profile 页面并行请求 5 个接口：`/api/students/me`、`/api/comments`、`/api/milestones`、`/api/activities`、`/api/scores`。  
重构后：
- **Dashboard** 只请求 `/api/students/me`（获取今日心情状态）+ `/api/mood`（获取本周记录）+ `/api/milestones`（最新 3 条）+ `/api/comments`（最新 1 条）
- **Profile** 使用一个聚合 API `/api/students/me/profile` 返回完整档案数据（减少并行请求数）

### 7.2 新增 API

| API | 方法 | 说明 |
|-----|------|------|
| `/api/students/me/profile` | GET | 聚合接口：学生信息 + 五维雷达 + 全部评语 + 全部里程碑 + 全部活动 + 成绩摘要 |
| `/api/students/me/todos` | GET | 返回今日待办列表（由后端根据打卡状态计算） |
| `/api/mood/stats` | GET | 返回心情统计：本月平均、最高、最低、积极天数、低落天数、连续记录天数 |
