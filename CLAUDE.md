# 新府学「一生一案」—— Claude Code 项目指南

## 项目概述

**「一生一案」** 是新府学私立学校的学生全周期成长档案与生涯发展平台。系统记录学生从入学到毕业的完整成长轨迹，整合学业成绩、心理测评、职业规划、性格分析、教师评语、活动参与、竞赛荣誉等多维数据，形成每位学生的专属生涯档案。

- **学校规模**：中型私立学校，1000~2000名学生
- **服务对象**：学生（主体）、教师（记录者）、家长（观察者）、管理者（决策者）
- **部署方式**：单台轻量级云服务器（4C8G~8C16G），Docker Compose 一键部署
- **开发方式**：VibeCoding——快速迭代、代码即设计、直觉驱动
- **AI 能力**：接入 DeepSeek 大模型，提供智能评语辅助、心理预警分析、职业规划建议

## 核心理念

### 范式转换

```
静态档案（成绩、评语）
    ↓
动态成长（五维雷达变化、里程碑时间线）
    ↓
自我认知（性格测评、职业倾向、心理状态）
    ↓
未来规划（AI辅助目标设定、路径推荐、资源匹配）
```

从"被评价的客体" → "主动经营自己成长的主体"。

### 双循环成长模型

| 层级 | 周期 | 载体 | 核心问题 |
|------|------|------|----------|
| **小循环** | 单次行为 | 课堂表现、作业、测验 | 我刚才的表现有反馈吗？ |
| **中循环** | 周/月 | 周反思、月度心理测评、活动参与 | 这个月我进步了吗？ |
| **大循环** | 学期/学年 | 五维雷达变化、里程碑、学期档案 | 我 overall 在成长吗？ |

### 五维雷达

```
            学业能力
           /    |    \
          /     |     \
    特长发展 ───┼─── 心理健康
          \     |     /
           \    |    /
            社交协作
```

| 维度 | 评估指标 | 数据来源 |
|------|----------|----------|
| **学业能力** | 成绩稳定性、学科均衡度、学习投入 | 成绩系统 + 教师评价 |
| **心理健康** | 情绪稳定性、压力应对、自我认知 | 心理测评 + 心情记录 |
| **职业规划** | 目标清晰度、职业兴趣匹配、行动力 | 职业规划测评 + 目标设定 |
| **社交协作** | 团队合作、沟通能力、人际关系 | 教师评语 + 活动表现 |
| **特长发展** | 兴趣深度、技能水平、成果产出 | 活动记录 + 比赛成绩 |

## VibeCoding 开发原则

本项目采用 **VibeCoding** 方式开发——以直觉和体验驱动，追求快速迭代和开发愉悦感。

### 核心信条

1. **代码即设计** —— 不写冗长的设计文档，用原型和代码表达设计意图
2. **先跑起来，再变优雅** —— 优先实现端到端功能链路，再重构优化
3. **删除而非添加** —— 能简化的绝不复杂化，没有明确价值的特性不做
4. **类型即契约** —— 全栈 TypeScript，用类型系统代替部分文档
5. **AI 辅助编码** —— 充分利用 Claude Code 和 DeepSeek 加速开发

### 实践准则

- **单体式优先**：拒绝微服务诱惑，所有功能在一个代码库中
- **约定优于配置**：使用框架默认值，不自定义除非必要
- **文件即模块**：一个文件一个核心概念，不提前抽象
- **数据库即真相**：Prisma schema 是数据模型的唯一来源
- **API 即函数**：API 路由是服务端函数的 HTTP 暴露，业务逻辑写在 service 层

## 技术架构

### 技术栈选型（轻量级全栈）

| 层级 | 技术 | 选型理由 |
|------|------|----------|
| **全栈框架** | **Next.js 15** (App Router) | 前后端一体、API Routes、React Server Components，减少技术栈数量 |
| **语言** | **TypeScript** | 全栈类型安全，开发时即发现错误 |
| **数据库** | **PostgreSQL 15** | 关系型数据为主，支持 JSON 字段灵活扩展 |
| **ORM** | **Prisma** | 类型安全、迁移管理、自动生成客户端 |
| **认证** | **NextAuth.js v5 (Auth.js)** | 与 Next.js 深度集成，支持多身份源 |
| **UI 组件** | **shadcn/ui** + **Tailwind CSS** | 即用即取、不依赖特定组件库版本、完全可控 |
| **状态管理** | **Zustand** | 轻量、无样板代码、TypeScript 友好 |
| **图表** | **Recharts** | React 原生、配置简单、足够满足雷达图和趋势图 |
| **文件存储** | **本地文件系统** / **MinIO** | 轻量级，初期可用本地存储，后期无缝切换 MinIO |
| **缓存** | **Next.js Cache** + **Redis (可选)** | 框架内置缓存优先，需要时引入 Redis |
| **AI 服务** | **DeepSeek API** | 国产大模型，性价比高，适合教育场景 |
| **部署** | **Docker Compose** | 单服务器一键部署，开发环境与生产环境一致 |

### 为什么选择 Next.js 全栈而非分离架构？

| 对比项 | Next.js 全栈 | 前后端分离 |
|--------|-------------|-----------|
| **项目数量** | 1 个 | 至少 3 个（前端×3 + 后端） |
| **类型共享** | 直接共享 | 需单独维护 types 包 |
| **API 定义** | 文件即路由 | 需维护 OpenAPI/Swagger |
| **部署复杂度** | Docker Compose 一键 | 需配置多个服务 |
| **开发体验** | 一个命令启动 | 需同时启多个服务 |
| **服务器资源** | 单进程服务 | 多个服务竞争资源 |

对于 1000~2000 学生的中型学校，全栈 Next.js 足以承载并发，且开发和运维成本最低。

## DeepSeek AI 集成方案

### 应用场景

| 场景 | 功能描述 | DeepSeek 模型 |
|------|----------|--------------|
| **AI 评语助手** | 教师输入关键词，AI 生成完整评语草稿 | deepseek-chat |
| **智能心理预警** | 分析心理测评文本，识别风险信号 | deepseek-chat |
| **个性化成长建议** | 基于五维雷达数据，生成改进建议 | deepseek-chat |
| **职业规划分析** | 分析测评结果，推荐职业方向 | deepseek-chat |
| **学期档案摘要** | 自动生成学期总结的文字描述 | deepseek-chat |
| **家长报告生成** | 将数据转化为家长友好的叙述 | deepseek-chat |
| **AI 问答助手** | 学生/教师的校园政策问答助手 | deepseek-chat |

### 架构设计

```
用户输入 → API Route (/api/ai/xxx) → DeepSeek 服务层 → DeepSeek API
                                              ↓
                                        结果后处理
                                              ↓
                                        返回前端
```

### 关键实现要点

1. **统一封装**：`lib/deepseek.ts` 封装所有 DeepSeek 调用，统一处理错误和重试
2. **流式输出**：AI 评语生成使用 SSE 流式返回，提升体验
3. **提示词模板**：提示词集中管理在 `lib/prompts/` 目录，版本化控制
4. **上下文管理**：职业规划等需要历史上下文的场景，使用数据库维护对话状态
5. **成本控制**：设置 Token 上限，非必要场景使用缓存结果
6. **内容安全**：AI 输出需经过敏感词过滤，教育场景尤其注意心理相关内容

### 提示词设计原则

- **角色明确**：每个提示词开头定义 AI 角色（如"你是一位资深中学心理教师"）
- **约束清晰**：明确输出格式、长度、风格要求
- **教育语境**：所有输出需符合中国教育场景和价值观
- **隐私保护**：提示词中不传输学生真实姓名和敏感信息，使用脱敏数据

## 数据库设计要点

### 规模估算（1500 学生规模）

| 数据项 | 估算 | 说明 |
|--------|------|------|
| 学生 | 1,500 | 在校生 |
| 教师 | 150 | 含班主任、学科教师、心理/职业规划老师 |
| 班级 | 45 | 每班约 35 人 |
| 成绩记录 | 27,000/学期 | 1500 学生 × 6 科目 × 3 次考试 |
| 评语 | 18,000/学期 | 1500 学生 × 12 条 |
| 心理测评 | 3,000/年 | 1500 学生 × 2 次 |
| 里程碑 | 4,500/年 | 每生平均 3 个 |
| 活动记录 | 7,500/年 | 每生平均 5 个 |

### 性能优化策略

- **索引策略**：student_id + created_at 复合索引覆盖 80% 查询
- **分页**：所有列表接口默认分页（page + pageSize）
- **聚合预计算**：班级五维雷达等聚合数据，用定时任务或触发器预计算
- **图片压缩**：上传图片自动压缩，缩略图与原图分离存储
- **连接池**：Prisma 连接池配置为 10~20，适配轻量级服务器

### 隐私分级

| 信息类型 | 可见范围 | 存储加密 |
|----------|----------|----------|
| 成绩 | 学生 + 教师 + 家长 | 否 |
| 教师评语 | 学生 + 对应教师 + 班主任 + 家长 | 否 |
| 心理测评详情 | 学生 + 心理老师 | **是** |
| 心理测评概览 | 学生 + 家长（脱敏） | 否 |
| 职业规划 | 学生 + 职业规划老师 | 否 |

## 三端设计

### 学生端 —— "看见自己"

核心功能：
- 📊 五维雷达（个人 + 班级平均对比 + 历史变化）
- 🎯 目标设定与追踪
- 📅 里程碑时间线
- 📝 查看教师评语（可收藏）
- 🧠 心理测评与心情日记
- 🎓 职业规划测评
- 📈 成绩趋势图
- 📋 学期档案

核心功能：
- 📋 我的班级 / 我的学生
- ⚠️ 心理预警列表（基于测评和心情数据）

核心功能：
- 学生/教师/班级 CRUD
- 测评量表配置
- 评语模板库维护
- 全校/年级/班级数据报表
- 里程碑规则配置
- 系统设置（学期时间、积分规则、隐私策略）
- 内容审核（学生申报里程碑）

## 开发规范

### 代码风格

- **TypeScript 严格模式**：开启 `strict: true`
- **函数优先**：优先使用函数组件和函数，少用 class
- **命名**：
  - 组件：PascalCase（`RadarChart.tsx`）
  - 工具函数：camelCase（`formatDate.ts`）
  - API 路由：kebab-case（`semester-reports/route.ts`）
  - 数据库模型：PascalCase（`CareerProfile`）
- **文件组织**：相关文件就近放置，不追求过深的目录层级

### API 设计

```
/api/v1
├── /auth               # Auth.js 处理
├── /students           # 学生 CRUD + 档案
├── /teachers           # 教师 + 我的学生/班级
├── /comments           # 评语（POST/GET/PUT）
├── /assessments        # 测评（提交/列表/结果）
├── /milestones         # 里程碑（创建/列表/审核）
├── /scores             # 成绩（录入/列表/趋势）
├── /activities         # 活动（记录/列表/审核）
├── /semester-reports   # 学期档案（生成/查看/发布）
├── /ai                 # DeepSeek AI 接口
│   ├── /comment-assist     # 评语辅助
│   ├── /psych-analysis     # 心理分析
│   ├── /career-suggest     # 职业建议
│   └── /report-summary     # 报告摘要
└── /admin              # 管理接口
```

### 权限检查模式

```typescript
// 在每个需要权限的 API Route 中使用
import { requireRole } from '@/lib/permissions';

// 教师端 API
export async function POST(req: Request) {
  const session = await requireRole(['teacher', 'admin']);
  // ...
}

// 学生只能访问自己的数据
export async function GET(req: Request, { params }: { params: { id: string } }) {
  const session = await requireAuth();
  if (session.user.role === 'student' && session.user.studentId !== params.id) {
    return new Response('Forbidden', { status: 403 });
  }
  // ...
}
```

## 关键约束与决策

### 做这个，不做那个

| ✅ 做 | ❌ 不做 |
|-------|--------|
| 单体式全栈 Next.js | 微服务架构 |
| PostgreSQL 关系型数据 | MongoDB 文档数据库 |
| 服务器渲染优先（RSC） | 纯 SPA + 独立后端 |
| AI 辅助生成（可控） | AI 自动发布（不可控） |
| 渐进增强（基础功能无 AI 也能用） | AI 依赖（没有 AI 系统瘫痪） |
| Docker Compose 单机部署 | Kubernetes 集群 |
| 学校内网/私有云部署 | 纯公有云 SaaS |

### 风险缓解

| 风险 | 缓解措施 |
|------|----------|
| 教师抵触增加工作量 | 评语 AI 辅助、快捷模板、操作极简 |
| 学生不关注档案 | 成长空间游戏化、班会课引导、与评优挂钩 |
| AI 生成内容不当 | 提示词严格约束、人工审核、敏感词过滤 |
| DeepSeek API 不可用 | 所有 AI 功能为增强型，核心功能不依赖 AI |
| 数据泄露 | 敏感数据加密、最小权限原则、操作审计日志 |
| 服务器资源不足 | 图片压缩、分页加载、聚合预计算、CDN 加速静态资源 |
